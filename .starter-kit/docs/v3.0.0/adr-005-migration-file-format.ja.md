# ADR-005: マイグレーションファイル形式の単一化（`.ts` 廃止・`.sql` + `.mjs`）

## ステータス

採択（v3.0.0）。ADR-001「Aurora DSQL + Drizzle ORM + カスタムマイグレーションランナー」の
「バッチデータ操作用の `.ts` マイグレーションファイルをサポート」という決定を supersede する。

## コンテキスト

v3 初期のマイグレーションランナーは `migrations/` 配下で `.sql` / `.ts` / `.mjs` の 3 形式を受け付け、
バッチデータ移行（テーブル再作成や 3,000 行超のバックフィル）は `.ts` で記述できた。

この設計には構造的な欠陥があった:

1. **local と Lambda で実行ファイルが分岐する。** ランナーはローカル開発では `tsx` で `.ts` を、
   Lambda（Docker, Node ランタイム）では `.mjs` を実行する想定だった。README も「Lambda では `.ts` を
   Dockerfile で `.mjs` に事前トランスパイルする」と記していたが、実際の migrator Dockerfile は
   `migrations/` を生コピーするだけでトランスパイル工程を持たず、記述と実装が乖離していた。
   結果、`.ts` マイグレーションは Lambda の Node ランタイムが `import()` した時点で SyntaxError となる。

2. **同一マイグレーションが 2 形式で併存しうる。** これは二重実行の温床になる。数ヶ月の実運用を経た
   ダウンストリームリポジトリでは、この分岐に対処するために「`_migrations` 台帳の重複実行防止キーを
   拡張子除去する（`WHERE name = $1 OR name = $2`）」「Docker ビルド内で `.ts`→`.mjs` に変換して `.ts` を
   削除する」機構を導入していた。これらは `.ts` を成立させるための正当なコストだが、大半のキット利用者が
   書く `.sql` と少数の `.mjs` には不要な複雑さである。

根本原因は「同一マイグレーションが `.ts`（local）と `.mjs`（Lambda）の 2 形式で存在しうる」ことである。
既定のランナーでは、コミットしたファイルがそのまま local と Lambda の双方で**同一に実行される**べきで、
トランスパイル工程を挟むべきではない。ただし移行が大規模・多ファイル化して型安全な記述が実質的に
リスクを下げる段階に至った利用者のために、`.ts` を封じるのではなく後述の逃げ道として位置づける。

## 決定

`migrations/` の**既定の正準形式を `.sql`（スキーマ変更）と `.mjs`（バッチデータ移行）の 2 つに限定**する。
既定のランナーは `.ts` を扱わない（`.mjs` を超える複雑さに達した場合の逃げ道は「逃げ道」節を参照）。

- ランナーの対象拡張子と zip asset への同梱を、すべて `.sql`/`.mjs` に揃える。トランスパイル工程は持たない。NodejsFunction bundle は `migrations/` を無変換でコピーし、`.mjs` をそのまま実行する。
- `.mjs` は `node`/`tsx` のいずれでも無変換で `import()` でき、ローカルと Lambda が同一ファイルを実行する。
  型補完は JSDoc（`/** @param {import('pg').PoolClient} client */`）で確保する。
- **`.mjs` のインタフェースは `export default async function(client, context)`。** 第 2 引数の `context`
  （`MigrationContext`）は、破壊的変更の前に S3 へバックアップを取るなど AWS リソースへアクセスする
  データ移行のための注入 seam である。既定は空で、サンプルアプリは何も必要としない。リソースを渡すには
  `MigrationContext` にフィールドを追加し、`migrate-cli.ts`（ローカル）と migrator handler（Lambda）で
  値を詰め、`DsqlMigrator` construct で対応する IAM を付与する。`context` を使わない移行は第 2 引数を
  無視してよい（`function(client)` のままで有効）。
- ランナーは `.sql` に対し実行時にも `transformSql` を適用する（`generate` を経由しない手書き SQL への多層防御）。

## 逃げ道: `.mjs` を超えたときの `.ts` 採用

移行が大規模・多ファイル・検証重視（例: 破壊的リネーム前に全表を S3 へダンプし SHA-256 で照合してから
変換する）になり、JSDoc 付き `.mjs` では型安全性・保守性が実質的に不足する段階に達したら、`.ts` の採用は
アンチパターンではなく正当な選択である。その場合の推奨形は次のとおり:

- **`.ts` を単一のコミット対象ソースとする。** `migrations/NNNN_x.ts` を 1 本だけコミットし、Docker ビルド内で
  esbuild により `.mjs` へトランスパイルして `.ts` はイメージから削除する（Lambda は `.mjs` のみを実行）。
  ローカルは `tsx` が `.ts` を直接実行する。複数ファイルに分けたい固有ロジックは `migrations/NNNN_x/` 配下へ置き、
  entrypoint から相対 import して `--bundle` で親 `.mjs` にインライン化する。
- **`_migrations` 台帳キーを拡張子非依存にする。** ローカルは `.ts`、Lambda は `.mjs` という形式分岐が残るため、
  同一移行を二重実行しないよう台帳の重複判定キーを拡張子除去する（`file.replace(/\.(sql|ts|mjs)$/, '')`）。
  ランナーの対象拡張子と CDK migrator のハッシュ対象にも `.ts` を含める。
- **`migrations/` 外への相対 import・`@repo/**`・`drizzle-orm`/`pg` の value import を禁止する\*\*（Lambda ランタイムで
  resolve できないため）。lint で境界を機械的に検査するとよい。

このコスト（形式分岐 + 拡張子除去キー + import 境界）を受け入れる判断をしたときにのみ導入する。既定のランナーに
これらを最初から組み込まないのは、大半の利用者（`.sql` 中心 + 稀な `.mjs`）に不要な複雑さを恒久的に課さないためである。

### 却下した代替案

- **`.ts` をトランスパイルして生成 `.mjs` を両方コミットする（authoring-only 方式）:** 実行時形式は `.mjs` 単一に
  なるが、`.ts`（ソース）と `.mjs`（生成物）の 2 ファイルがコミット対象として併存し、手編集などで乖離するリスクが
  大きい。単一ソースを保つ上記の逃げ道（`.ts` のみコミット）を優先する。
- **`.ts` を無変換で Lambda 実行する**（`tsx` をイメージに同梱する / Node の `--experimental-strip-types` を使う）:
  前者は実行時依存とローダ登録が増え、イメージも肥大化する。後者は experimental フラグ依存で、Node の
  マイナーバージョン差や警告出力のリスクがある。いずれも「copy して育てる」キットの再現性重視に反する。

## 結果

- **`.sql` マイグレーションと既存の `.mjs` 利用者には影響がない。** 大半のマイグレーションは drizzle-kit が
  生成する `.sql` であり、`.mjs` は稀なバッチデータ移行に限られるため、影響範囲は限定的。
- **既定のランナー**は形式分岐を持たず、`_migrations` 台帳キーはフルファイル名のままで曖昧さがない
  （1 マイグレーション = 1 ファイル 1 拡張子）。拡張子除去による重複実行防止キーは、逃げ道として `.ts` を
  導入しない限り不要である。
- **`context` 注入 seam により、破壊的変更前の外部バックアップなど AWS リソースを要するデータ移行を、
  ファイル形式を変えずに `.mjs` のまま実装できる。** 移行安全性（形式非依存の冪等・swap 前検証・外部バックアップ）は
  ファイル形式ではなくこの seam と各移行の書き方が担う。
- ドキュメント（`packages/db/README.md`, `AGENTS.md`）と実装（Dockerfile が生コピーである事実）が一致する。

### 破壊的変更: 既存 v3 利用者向け移行手順

既に `.ts` でバッチデータ移行を書いている場合、以下の移行が必要:

1. `migrations/NNNN_x.ts` を `migrations/NNNN_x.mjs` にリネームする。
2. TypeScript の型注釈を JSDoc に置き換える（`export default async function(client)` の形は維持。
   `import type { PoolClient }` は不要で、`/** @param {import('pg').PoolClient} client */` を用いる）。
3. **未適用**のマイグレーションであれば、これで完了。
4. **既に適用済み**のマイグレーションの場合は注意が必要。`_migrations` テーブルには旧名 `NNNN_x.ts` が
   記録されている。ランナーはフルファイル名でスキップ判定するため、リネーム後の `NNNN_x.mjs` は
   **未適用とみなされ再実行される**。データ移行の二重適用を避けるため、デプロイ前に各環境で次のいずれかを行う:
   - 台帳の名前を更新する: `UPDATE _migrations SET name = 'NNNN_x.mjs' WHERE name = 'NNNN_x.ts';`
   - あるいはマイグレーションを冪等に書き直し、再実行されても安全にする。

本キットは copy-and-grow 型で早期利用者数が限定的なため、拡張子を無視する後方互換キー
（ダウンストリームが採用したもの）はランナーに導入せず、上記の明示的な移行手順を採る判断とした。
ランナーを単純に保つこと、および `_migrations` の意味論（記録されたファイル名 = 適用済み）の明快さを優先する。
