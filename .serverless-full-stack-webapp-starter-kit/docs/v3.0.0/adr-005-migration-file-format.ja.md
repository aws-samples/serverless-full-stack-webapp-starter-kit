# ADR-005: マイグレーションファイル形式の単一化（`.ts` 廃止・`.sql` + `.mjs`）

## ステータス

採択（v3.0.0）。ADR-001「Aurora DSQL + Drizzle ORM + カスタムマイグレーションランナー」の
「バッチデータ操作用の `.ts` マイグレーションファイルをサポート」という決定を supersede する。

## コンテキスト

v3 初期のマイグレーションランナーは `migrations/` 配下で `.sql` / `.ts` / `.mjs` の 3 形式を受け付け、
バッチデータ移行（テーブル再作成や 3,000 行超のバックフィル）は `.ts` で記述できた。

この設計には構造的な欠陥があった:

1. **local と Lambda で実行ファイルが分岐する。** ランナーはローカル開発では `tsx` で `.ts` を、
   Lambda（Docker, Node ランタイム）では `.mjs` を実行する想定だった。README も「Lambda では `.ts` を
   Dockerfile で `.mjs` に事前トランスパイルする」と記していたが、実際の migrator Dockerfile は
   `migrations/` を生コピーするだけでトランスパイル工程を持たず、記述と実装が乖離していた。
   結果、`.ts` マイグレーションは Lambda の Node ランタイムが `import()` した時点で SyntaxError となる。

2. **同一マイグレーションが 2 形式で併存しうる。** これは二重実行の温床になる。数ヶ月の実運用を経た
   ダウンストリームリポジトリでは、この分岐に対処するために「`_migrations` 台帳の重複実行防止キーを
   拡張子除去する（`WHERE name = $1 OR name = $2`）」「Docker ビルド内で `.ts`→`.mjs` に変換して `.ts` を
   削除する」といった**対症療法**を積み増す結果になった。

根本原因は「同一マイグレーションが `.ts`（local）と `.mjs`（Lambda）の 2 形式で存在しうる」ことである。
マイグレーションは、コミットしたファイルがそのまま local と Lambda の双方で**同一に実行される**べきで、
トランスパイル工程を挟むべきではない。

## 決定

`migrations/` の正準形式を **`.sql`（スキーマ変更）と `.mjs`（バッチデータ移行）の 2 つに限定**し、
`.ts` を**非サポート**とする。

- ランナーの対象拡張子、CDK migrator のマイグレーション内容ハッシュ、Docker への同梱を、すべて
  `.sql`/`.mjs` に揃える。トランスパイル工程は持たない（Docker は `migrations/` を生コピーし、`.mjs` を
  そのまま実行）。
- `.mjs` は `node`/`tsx` のいずれでも無変換で `import()` でき、ローカルと Lambda が同一ファイルを実行する。
  型補完は JSDoc（`/** @param {import('pg').PoolClient} client */`）で確保する。`export default async function(client)`
  というインタフェースは従来どおり。
- ランナーは `.sql` に対し実行時にも `transformSql` を適用する（`generate` を経由しない手書き SQL への多層防御）。

### 却下した代替案

- **Docker ビルド内で `.ts`→`.mjs` にトランスパイルする**（ダウンストリームで採られた方式）:
  local=`.ts` / Lambda=`.mjs` という形式分岐が残るため、二重実行を防ぐための拡張子除去キーなどの複雑さを
  恒久的に抱え続ける。根本原因（2 形式併存）を解消しない対症療法である。
- **`.ts` を無変換で Lambda 実行する**（`tsx` をイメージに同梱する / Node の `--experimental-strip-types` を使う）:
  前者は実行時依存とローダ登録が増え、イメージも肥大化する。後者は experimental フラグ依存で、Node の
  マイナーバージョン差や警告出力のリスクがある。いずれも「copy して育てる」キットの再現性重視に反する。
  マイグレーションでの型安全を最優先する要件が将来生じた場合にのみ再検討する。

## 結果

- **`.sql` マイグレーションと既存の `.mjs` 利用者には影響がない。** 大半のマイグレーションは drizzle-kit が
  生成する `.sql` であり、`.mjs` は稀なバッチデータ移行に限られるため、影響範囲は限定的。
- ランナーは形式分岐を持たなくなり、`_migrations` 台帳キーはフルファイル名のままで曖昧さがない
  （1 マイグレーション = 1 ファイル 1 拡張子）。拡張子除去による重複実行防止キーは不要になった。
- ドキュメント（`packages/db/README.md`, `AGENTS.md`）と実装（Dockerfile が生コピーである事実）が一致する。

### 破壊的変更: 既存 v3 利用者向け移行手順

既に `.ts` でバッチデータ移行を書いている場合、以下の移行が必要:

1. `migrations/NNNN_x.ts` を `migrations/NNNN_x.mjs` にリネームする。
2. TypeScript の型注釈を JSDoc に置き換える（`export default async function(client)` の形は維持。
   `import type { PoolClient }` は不要で、`/** @param {import('pg').PoolClient} client */` を用いる）。
3. **未適用**のマイグレーションであれば、これで完了。
4. **既に適用済み**のマイグレーションの場合は注意が必要。`_migrations` テーブルには旧名 `NNNN_x.ts` が
   記録されている。ランナーはフルファイル名でスキップ判定するため、リネーム後の `NNNN_x.mjs` は
   **未適用とみなされ再実行される**。データ移行の二重適用を避けるため、デプロイ前に各環境で次のいずれかを行う:
   - 台帳の名前を更新する: `UPDATE _migrations SET name = 'NNNN_x.mjs' WHERE name = 'NNNN_x.ts';`
   - あるいはマイグレーションを冪等に書き直し、再実行されても安全にする。

本キットは copy-and-grow 型で早期利用者数が限定的なため、拡張子を無視する後方互換キー
（ダウンストリームが採用したもの）はランナーに導入せず、上記の明示的な移行手順を採る判断とした。
ランナーを単純に保つこと、および `_migrations` の意味論（記録されたファイル名 = 適用済み）の明快さを優先する。
